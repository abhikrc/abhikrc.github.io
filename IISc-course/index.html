Rationale for the Course:

This course covers different techniques to guarantee reliability of software. Developing reliable software is a difficult task which most Computer Science professionals have to face. Furthermore, reliability guarantees are particularly important for safety critical software. This course will familiarize our graduate students with some techniques which can help in this regard. The course is very timely with the increased role of software in controlling devices of everyday use (popularly known as embedded systems).

Brief Description

The immense growth in the complexity of software has increased the scope of errors, which are often critical. The nature of these 
errors is diverse, resulting from the diversity of the various classes of software: sequential, multithreaded, reactive and real-time.  
In this course, we will study techniques for verification, run-time monitoring and debugging of software which help us to give certain 
guarantees against such errors. The focus will be on automated validation techniques.

 SPIN model checker SPIN has its own modeling language (called PROMELA) which captures some of the features of a programming language; actually PROMELA is suitable for directly modeling protocols. We will discuss PROMELA in the first  lecture.

 We will then continue with our discussion of the verification tool SPIN.  We will discuss its capabilities for model checking, random simulation and user-guided simulation. The Linear-time temporal logic (LTL) model checking algorithm implemented in SPIN is discussed.

Week 8

 2 lectures

 Testing Testing is the oldest form of software validation and still constitutes a large part of the state-of-practice. We will discuss test coverage criteria in this lecture. Some recently investigated criteria such as the MC/DC criteria which is important for software certification in safety-critical industries (such as avionics, and to some extent automobile software) will be discussed. We will then discuss the integration of testing and model checking

Debugging:  Program testing allows us to identify the problematic test inputs i.e. test cases where the "output" does not match programmer's expectations. However, once the problematic tests are identified how do we analyze the program to locate the actual bug? We will cover this topic in the last lecture. We will start with well-known methods like slicing and move to newer methods which proceed by comparison of traces for different test cases.

 Readings on testing:  Chapter 9 of the book "Software Reliability Methods" by Doron Peled. Also see
